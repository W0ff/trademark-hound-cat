## Standard Architecture

### System Overview

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        INVOCATION LAYER                                  │
│                                                                          │
│   User runs: /trademark-cat ACME "software company"                     │
│   User runs: /trademark-hound ACME                                      │
└──────────────────────────┬──────────────────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────────────────┐
│                     SKILL 1: TRADEMARK CAT                               │
│                                                                          │
│  .claude/skills/trademark-cat/SKILL.md                                  │
│  ┌────────────────────────────────────────────────────────────────────┐ │
│  │  1. Accept: trademark name + company context                       │ │
│  │  2. Generate ~100 variants across 5 linguistic categories          │ │
│  │  3. Present variants for user review (interactive loop)            │ │
│  │  4. Iterate on feedback until approved                             │ │
│  │  5. Write output: variants-[TRADEMARK].txt                         │ │
│  └────────────────────────────────────────────────────────────────────┘ │
└──────────────────────────┬──────────────────────────────────────────────┘
                           │ writes
                           ▼
┌──────────────────────────────────────────────────────────────────────────┐
│                        ARTIFACT LAYER (workspace files)                  │
│                                                                          │
│   variants-[TRADEMARK].txt        ← variants from Cat, consumed by Hound │
│   safelist-[TRADEMARK].json       ← persists across Hound runs           │
│   search-[TRADEMARK].py           ← generated Serper.dev script (temp)   │
│   leads-[TRADEMARK].json          ← raw search hits from Python script   │
│   report-[TRADEMARK]-[DATE].md    ← final output, reviewed offline       │
└──────────────────────────────────────────────────────────────────────────┘
                           │ reads
                           ▼
┌──────────────────────────────────────────────────────────────────────────┐
│                     SKILL 2: TRADEMARK HOUND                             │
│                                                                          │
│  .claude/skills/trademark-hound/SKILL.md                                │
│  ┌────────────────────────────────────────────────────────────────────┐ │
│  │  A. GUARD: Check for variants file, route to Cat if missing        │ │
│  │  B. LOAD: Read safelist-[TRADEMARK].json (empty set if none)       │ │
│  │  C. SEARCH: Generate + run Serper.dev Python script                │ │
│  │  D. FILTER: Remove safe-listed domains/URLs from leads             │ │
│  │  E. INVESTIGATE: Agentic browsing per unseen lead                  │ │
│  │  F. SCORE: Apply 8-factor weighted threat matrix                   │ │
│  │  G. REPORT: Write dated Markdown (Medium+High only)                │ │
│  │  H. SAFE LIST UPDATE: Accept reviewed report, update JSON          │ │
│  └────────────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────────────┘
```

### Component Responsibilities

| Component | Responsibility | Implementation |
|-----------|----------------|----------------|
| **Trademark Cat skill** | Variant generation and user approval loop | `SKILL.md` prompt instructions; Claude holds conversation loop |
| **variants-[TM].txt** | Hand-off contract between skills | Plain text file, one variant per line, written by Cat, read by Hound |
| **Trademark Hound skill** | Orchestration of the full investigation pipeline | `SKILL.md` with multi-step instructions; calls Python sub-script |
| **Serper.dev Python script** | Batch SERP queries via web API | Generated by Claude, written to workspace, executed via Bash tool |
| **leads-[TM].json** | Intermediate state: raw search hits | Written by Python script, read back by Hound for filtering |
| **WebFetch/Bash browsing** | Agentic investigation of each lead URL | Claude's built-in tools; no separate component needed |
| **Threat scoring logic** | 8-factor matrix, produces 0–48 integer per lead | Embedded as instructions in Hound `SKILL.md`; Claude applies inline |
| **safelist-[TM].json** | Persistent false-positive elimination across runs | Read at startup, updated when reviewed report is passed as argument |
| **Report Markdown** | Dated, reviewable output for attorneys | Written by Claude to workspace via Write tool |

## Recommended Project Structure

```
.claude/
├── skills/
│   ├── trademark-cat/
│   │   ├── SKILL.md               # Main instructions + $ARGUMENTS handling
│   │   └── examples/
│   │       └── variant-output.md  # Example of expected variant format
│   └── trademark-hound/
│       ├── SKILL.md               # Orchestration instructions
│       ├── scripts/
│       │   └── serper_search.py   # Template/skeleton for the Serper API script
│       └── examples/
│           └── report-sample.md   # Example report structure for Claude to follow

[workspace]/                        # Directory where user invokes skills
├── variants-ACME.txt              # Written by Cat, consumed by Hound
├── safelist-ACME.json             # Persists across all Hound runs for ACME
├── leads-ACME.json                # Intermediate: raw Serper.dev results
├── search-ACME.py                 # Generated Python script (can be deleted after run)
└── reports/
    └── report-ACME-2026-04-03.md  # Dated report, one per run
```

### Structure Rationale

- **.claude/skills/trademark-cat/ and trademark-hound/**: Separate skill directories per the Agent Skills standard. Each skill is independently invocable and self-contained.
- **scripts/ inside each skill**: Bundled scripts live with their skill so `${CLAUDE_SKILL_DIR}` can reference them reliably regardless of working directory.
- **Workspace artifacts at project root**: All runtime files (variants, safelist, leads, reports) live in the invoking workspace directory. This keeps attorney workspace organized by trademark name and makes files easy to locate and share.
- **reports/ subdirectory**: Keeps the workspace root clean; dated filenames allow historical comparison across monitoring cycles.

## Architectural Patterns

### Pattern 1: File-Contract Hand-off Between Skills

**What:** Upstream skill (Cat) writes a named file; downstream skill (Hound) reads it. The file is the entire interface between skills — no shared memory, no direct invocation chain.

**When to use:** When skills may be run days apart, by different users, or when the upstream output needs human review before the downstream run. Trademark Cat is designed to be run once per review cycle; Trademark Hound may be run multiple times against the same variant set.

**Trade-offs:** Simple and portable. Attorneys can inspect `variants-ACME.txt` before hounding. Downside: no enforcement that Cat runs first — Hound must guard for missing file.

**Guard pattern in Hound SKILL.md:**
```
1. Check if variants-$ARGUMENTS.txt exists in the current directory.
2. If it does not exist, inform the user and suggest running:
   /trademark-cat $ARGUMENTS "[company context]"
   Then stop.
3. If it exists, proceed with the investigation pipeline.
```

### Pattern 2: Claude-Generated Python Script for API Calls

**What:** Hound's `SKILL.md` instructs Claude to write a Python script using the bundled template, inject the user's Serper.dev API key, write it to the workspace, execute it via the Bash tool, and read back the JSON output.

**When to use:** When a web API call requires a Python environment (rate limiting, pagination, JSON parsing) that Claude cannot do natively. Serper.dev returns paginated JSON; a Python script handles this more robustly than inline Bash.

**Trade-offs:** The generated script is inspectable and re-runnable by the attorney. Adds a step (write → execute → read) but keeps the API key out of the `SKILL.md` instructions and allows the script to be audited. The script is disposable — it can be deleted after the run or kept for debugging.

**Invocation pattern in Hound SKILL.md:**
```
Generate search-$ARGUMENTS.py based on scripts/serper_search.py.
Substitute:
  - TRADEMARK with $ARGUMENTS
  - VARIANTS with the contents of variants-$ARGUMENTS.txt
  - API_KEY with the user's Serper.dev key (ask if not provided)
Write the script to the current directory.
Execute: python3 search-$ARGUMENTS.py
Read the output file: leads-$ARGUMENTS.json
```

### Pattern 3: Inline Agentic Browsing (No Subagent Needed)

**What:** Hound investigates each lead URL using Claude's built-in WebFetch tool inline — no forked subagent or separate skill. Investigation, scoring, and report accumulation happen in a single pass through the leads list.

**When to use:** When the lead count is bounded (typically 20–50 after filtering), the investigation is sequential by design (each score feeds the final sorted report), and parallelism adds complexity without proportional benefit.

**Trade-offs:** Simpler architecture and easier to reason about than a forked subagent pool. Slightly slower than parallel browsing for large lead sets, but trademark monitoring runs are not latency-sensitive (attorneys run them periodically, not in real-time).

**Upgrade path:** If lead counts grow significantly, wrap the investigation loop in `context: fork` with the Explore agent. For the initial build, inline is preferred.

### Pattern 4: JSON Safe List as Persistent State

**What:** `safelist-[TRADEMARK].json` stores domains and URLs that have been reviewed and confirmed as non-threats. Hound loads this file at startup and filters matching leads before investigation. Hound updates this file when a reviewed report (with THREAT? column filled) is passed as an argument.

**When to use:** Persistent state that must survive across sessions and be human-readable. A flat JSON file meets all requirements without database infrastructure.

**Trade-offs:** Simple, portable, and inspectable by attorneys. No merge conflicts in version control if attorneys work on separate trademarks. Limitation: no conflict resolution if two users update the same safelist concurrently — acceptable for the single-attorney-per-trademark workflow.

**File schema:**
```json
{
  "trademark": "ACME",
  "safe_entries": [
    {
      "url": "https://example.com/acme-cleaning",
      "domain": "example.com",
      "reason": "Unrelated industry - cleaning products",
      "added": "2026-04-03"
    }
  ]
}
```

## Data Flow

### Full Pipeline Flow

```
User invokes /trademark-cat ACME "software company"
    │
    ▼
Claude generates variants (5 linguistic categories × ~20 each)
    │
    ▼
Claude presents list, user provides feedback
    │
    ▼ (loop until approved)
Claude writes variants-ACME.txt
    │
    ▼
User invokes /trademark-hound ACME [serper-key]
    │
    ├── GUARD: variants-ACME.txt exists? → NO → stop, tell user to run Cat
    │                                     → YES → continue
    ├── LOAD: safelist-ACME.json → set of safe URLs/domains
    ├── GENERATE: write search-ACME.py from template + variants
    ├── EXECUTE: python3 search-ACME.py → leads-ACME.json (raw hits)
    ├── FILTER: remove safe-listed hits from leads
    ├── INVESTIGATE: WebFetch each remaining lead URL
    ├── SCORE: apply 8-factor matrix, compute 0–48 integer per lead
    ├── THRESHOLD: keep Medium (10–14) and High (≥15) only
    └── WRITE: report-ACME-2026-04-03.md
    │
    ▼
Attorney reviews report, fills THREAT? column
    │
    ▼
User invokes /trademark-hound ACME --update-safelist report-ACME-2026-04-03.md
    │
    ▼
Hound reads reviewed report, adds NO-THREAT entries to safelist-ACME.json
```

### Safe List Update Sub-flow

```
Reviewed report (with THREAT? = NO rows)
    │
    ▼
Hound reads report Markdown, extracts NO-THREAT rows
    │
    ▼
For each NO-THREAT entry: append to safelist-ACME.json safe_entries
    │
    ▼
Write updated safelist-ACME.json
    │
    ▼
Confirm count of entries added to user
```

### Key Data Flows

1. **Cat → Hound via file:** `variants-ACME.txt` is the only interface between skills. Cat writes it; Hound reads it. No other communication path exists.
2. **Hound → Python via generated script:** Hound generates a Python file, executes it as a subprocess, and reads back JSON output. The Python script is the only component making external HTTP calls.
3. **Hound → attorney via Markdown report:** The report is the external deliverable. It is designed for offline review, not for machine consumption.
4. **Attorney → Hound via reviewed report:** The only input back into the system after investigation is the reviewed Markdown file, which drives safe list updates.

## Build Order

Dependencies determine build order. Build in this sequence:

| Order | Component | Depends On | Rationale |
|-------|-----------|------------|-----------|
| 1 | Serper.dev Python script template | Nothing | Standalone; validates API key and search mechanics before any skill work |
| 2 | variants file format and safe list JSON schema | Nothing | Define the contracts before building the consumers |
| 3 | Trademark Cat skill | Variants file schema | Cat writes the file; schema must be settled first |
| 4 | Threat scoring matrix (8 factors, weightings) | Nothing | Define scoring before Hound is written; scoring logic is embedded in Hound's SKILL.md |
| 5 | Trademark Hound skill (search + filter + score + report) | Python template, variants schema, safe list schema, scoring matrix | All upstream dependencies must be stable |
| 6 | Safe list update sub-flow | Hound skill, report format | Update flow consumes the report format Hound produces; build after Hound |

## Anti-Patterns

### Anti-Pattern 1: Embedding the Serper API Key in SKILL.md

**What people do:** Put the API key directly in the skill instructions as a constant or environment variable reference.

**Why it's wrong:** SKILL.md files are typically committed to version control. An API key in a committed file will eventually be exposed.

**Do this instead:** Instruct Claude to ask the user for the API key at invocation time (`/trademark-hound ACME [api-key]`) and inject it only into the generated Python script at runtime. The script is a workspace file, not a tracked file, and can be gitignored.

### Anti-Pattern 2: Having Cat Directly Invoke Hound

**What people do:** At the end of Cat's instructions, add a step that calls `/trademark-hound` automatically.

**Why it's wrong:** Attorneys need to review and potentially edit the variants list before running a search. Automatic chaining removes this control point. It also means Cat and Hound become tightly coupled — Cat must know Hound's invocation signature.

**Do this instead:** Cat writes the file and clearly reports the file path. Hound's guard step validates the file exists and tells the user to run Cat if not. User invokes each skill independently.

### Anti-Pattern 3: One Monolithic Skill

**What people do:** Combine variant generation and investigation into a single skill to avoid the two-step invocation.

**Why it's wrong:** Variant generation is an interactive, iterative process. Investigation is a long-running batch process. Combining them means the attorney cannot approve variants before a potentially expensive search run starts. It also makes each skill impossible to use independently (e.g., re-running Hound with updated scoring after changing the matrix, without re-generating variants).

**Do this instead:** Maintain the two-skill separation. The file hand-off is the correct decoupling point.

### Anti-Pattern 4: Storing All Leads in the Report (Not in leads JSON)

**What people do:** Have the Python script write results directly to the Markdown report, skipping the intermediate JSON file.

**Why it's wrong:** The raw search results (including safe-listed and low-scoring leads) are useful for debugging. If Hound re-runs with different score thresholds, the raw leads should be available without re-querying Serper. JSON is also easier for Claude to parse programmatically than Markdown.

**Do this instead:** Python script writes `leads-[TM].json` (all raw hits). Hound filters and scores from JSON, writes only qualified leads to Markdown.

## Scaling Considerations

This is a CLI tool for periodic manual use by attorneys — scaling in the traditional sense does not apply. The relevant "scaling" questions are about portfolio size:

| Portfolio Scale | Architecture Adjustments |
|----------------|--------------------------|
| 1–10 trademarks | Current flat-file architecture is fine; all files in one workspace directory |
| 10–50 trademarks | Organize workspace by trademark subdirectory: `ACME/variants-ACME.txt`, `ACME/safelist-ACME.json`, etc. |
| 50+ trademarks | Consider a client-per-directory structure; safe lists may warrant a shared format for cross-trademark analysis |

### Scaling Priorities

1. **First friction point:** Finding files across many trademarks in a flat workspace directory. Fix: subdirectory-per-trademark organization (can be introduced in Phase 2 without breaking Phase 1 design).
2. **Second friction point:** Running Hound on many trademarks sequentially. Fix: a batch orchestration skill that loops over a list of trademarks — out of scope for initial build but enabled by the file-contract architecture.
