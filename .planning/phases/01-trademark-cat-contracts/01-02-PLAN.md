  <action>
Write `.claude/commands/trademark-cat.md` as a Claude Code slash command. This file IS the system prompt — Claude reads it verbatim when an attorney invokes `/trademark-cat`. Write it as clear, unambiguous instructions in second-person imperative prose.

The file MUST include YAML frontmatter, then the following sections in order:

**Frontmatter:**
```yaml
---
description: Generate ~100 confusion-risk trademark variants for attorney review. Use when preparing a trademark monitoring search or generating a variants file for /trademark-hound.
---
```

**Section: Intake**

Parse `$ARGUMENTS` to extract:
1. The trademark name (first token or quoted string)
2. The goods/services description (quoted string, or everything after the first token)

If `$ARGUMENTS` is empty or only contains a trademark name with no goods/services description, ask: "What goods or services does [TRADEMARK] cover? For example: 'software for project management' or 'retail clothing stores'."

Do not proceed until you have BOTH the trademark name and a goods/services description.

Sanitize the trademark name for use in filenames: replace spaces with hyphens, remove any characters that are not letters, numbers, or hyphens. Store the sanitized name for all file output. Example: "OMEGA 3" → "omega-3".

**Section: Negative Constraints**

From the goods/services description, derive a Negative Constraints (Ignore List): a list of non-competing industries that should be excluded from confusion analysis. Example: for "Delta" in "airlines," exclude dental, plumbing, carpentry, and financial services.

State these constraints clearly to yourself (do not present to the attorney) before generating variants.

**Section: Variant Generation**

Generate ~100 variants across these 5 categories, targeting approximately 20 per category:

1. Phonetic & Orthographic — sound-alike spellings, vowel swaps, consonant substitutions, homophones, alternate spellings (e.g., ACME → Ackme, Akme, ACMEE)
2. Compound Suffix-State — adding or removing common suffixes (-Tech, -Corp, -Pro, -Hub, -AI, -Labs, -Soft, -Co, -Group) and geographic/state combinations
3. Semantic Synonyms — words with equivalent or near-equivalent meaning to the trademark
4. Conceptual Variants — related concepts, translated meanings, associated ideas, category associations
5. Additional Phonetics/Hybrids — cross-category hybrids, phonetic transliterations, stylistic variations combining approaches from multiple categories

After generating, COUNT the variants per category. If any category has fewer than 15 variants, generate additional variants for that category before presenting. Do not present the list until all 5 categories have at least 15 variants.

Variant names MUST NOT contain the `#` character. Variant names may only contain letters, numbers, hyphens, spaces, and common punctuation.

For each variant, internally record:
- The confusion axis: one of `phonetic`, `visual`, `conceptual`, or `compound`
- A one-sentence rationale (e.g., "vowel swap; sounds near-identical when spoken aloud")

These annotations are NOT shown during the review loop — they are written to the file only.

**Section: Presentation and Approval Loop**

Present the variants using EXACTLY this format:

```
---
Total: [N] variants | Phonetic: [X] | Compound: [Y] | Semantic: [Z] | Conceptual: [W] | Hybrid: [V]

# Phonetic & Orthographic ([X])
[variant 1]
[variant 2]
...

# Compound Suffix-State ([Y])
[variant 1]
...

# Semantic Synonyms ([Z])
...

# Conceptual Variants ([W])
...

# Additional Phonetics/Hybrids ([V])
...
---
```

Show variant NAMES ONLY in the list — no comments, no annotations, no rationale. The `# Category (N)` headers are shown; inline annotations are NOT.

After presenting, tell the attorney:

> "Review this list. To give feedback — add variants, remove variants, rebalance categories, or anything else — just describe what you'd like changed in plain language. To approve and write the variants file, say 'approved' or 'looks good'."

**Watch for feedback:** Any instruction implying adding, removing, renaming, or rebalancing variants. If feedback is given, revise the list accordingly and present the FULL revised list again from the top. Do not show a diff — always show the complete current list.

**Watch for approval intent:** "approved," "looks good," "that's fine," "go ahead," "yes," "perfect," "ship it." If there is ANY ambiguity about whether the attorney is approving or giving feedback, ask: "Just to confirm — shall I write the approved variants file now?" Do NOT write the file until approval is explicit.

Repeat this loop until explicit approval is received.

**Section: Writing the Variants File**

After explicit attorney approval, construct the variants file content as follows:

Line 1: `# Context: [goods/services description as given by attorney]`

Then for each category, write:
```
# [Category Name]
[variant name]  # [confusion axis]: [one-sentence rationale]
```

Example:
```
# Context: software for project management
# Phonetic & Orthographic
Ackme  # phonetic: vowel shift; sounds near-identical when spoken aloud
A.C.M.E  # orthographic: punctuation insertion; phonetically identical
```

Write the file to the current working directory as: `variants-[sanitized-trademark-lowercase].txt`

Use the Write tool to create the file. After the Write tool confirms success, output exactly:

> "Variants file written to: variants-[trademark].txt ([N] variants across 5 categories)"

Do not write the file before explicit approval. Do not write the file more than once per invocation.
